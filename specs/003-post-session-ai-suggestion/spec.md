# Feature Specification: Post-Session AI Suggestion

**Feature Branch**: `003-post-session-ai-suggestion`  
**Created**: 2026-05-03  
**Status**: Draft  
**Input**: User description: "As a user I want to see a suggestion from AI chat after I save the session. At the moment the backend already has functionality to provide feedback after the session — it is just logged to the console right now."

**Language Rule**: This specification MUST be written in English.

## User Scenarios & Testing *(mandatory)*

### User Story 1 - See AI Suggestion After Saving a Program Session (Priority: P1)

After a user saves a program workout session, the system generates a personalised AI coaching suggestion and presents it to the user before they continue to the session history. The suggestion acknowledges what the user did in the session and provides actionable progression advice.

**Why this priority**: This is the core value of the feature — surfacing AI-generated coaching feedback that is already being generated by the backend but discarded. Without this, the rest of the feature has no starting point.

**Independent Test**: Can be tested end-to-end by logging a program session and observing that an AI suggestion panel appears on the confirmation/result screen, even before persistence or history integration is in place.

**Acceptance Scenarios**:

1. **Given** a user has filled in a program session form, **When** they submit (save) the session, **Then** after a brief loading period, an AI coaching suggestion is displayed on the screen before the user navigates away.
2. **Given** the AI service returns a suggestion, **When** the suggestion is displayed, **Then** it appears clearly separated from the user's session data (feelings, exercises) and is labelled as an AI-generated insight.
3. **Given** the AI service takes time to respond, **When** the session has been saved, **Then** the user sees a loading indicator in the suggestion area rather than a blank or broken layout.
4. **Given** the AI service fails to produce a suggestion, **When** the error occurs, **Then** the user is not blocked and the session detail is still accessible; the suggestion area either shows a fallback message or is absent.

---

### User Story 2 - AI Suggestion Persisted and Accessible in Session History (Priority: P2)

Once generated, the AI suggestion is saved alongside the session and can be viewed again when the user revisits that session in the history. The suggestion never changes after it is first stored.

**Why this priority**: Persistence is essential for the suggestion to have lasting value. Without it, the user only sees the suggestion once immediately after saving, and loses it on refresh or return visit.

**Independent Test**: Can be tested by: (1) saving a session and noting the suggestion text, (2) navigating away, (3) returning to the session detail in history and confirming the same suggestion text is still shown.

**Acceptance Scenarios**:

1. **Given** an AI suggestion was shown after session save, **When** the user later opens that session from the history list, **Then** the same suggestion text is displayed in the session detail view.
2. **Given** a suggestion was already stored for a session, **When** the user saves the session again (re-load scenario) or the AI produces a different result, **Then** the stored suggestion remains unchanged — it is never overwritten.
3. **Given** a session that has no suggestion (e.g., an older session saved before this feature, or a free session), **When** the user opens that session detail, **Then** the suggestion section is simply absent — no error and no placeholder text.

---

### User Story 3 - Free Sessions Do Not Generate AI Suggestions (Priority: P3)

The AI suggestion is scoped to program sessions only, since the underlying AI model analyses performance against program targets. Free sessions do not trigger suggestion generation.

**Why this priority**: The AI analysis logic in the backend already gates on `SessionType.PROGRAM`. This story specifies the intentional boundary so that the front-end and any future feature work do not attempt to show a suggestion for free sessions.

**Independent Test**: Can be tested by logging a free session and confirming no suggestion panel appears and no suggestion-related network call is made.

**Acceptance Scenarios**:

1. **Given** a user logs a free (non-program) session, **When** the session is saved, **Then** no AI suggestion is requested or displayed.
2. **Given** a free session in history, **When** the user views its detail, **Then** the suggestion section is not rendered.

---

### Edge Cases & Simplicity Checks

- **What if the AI response arrives after the user has already navigated away?** The suggestion is persisted asynchronously; it will be available the next time the user opens the session detail. No pop-up or push notification is sent.
- **What if the AI generates an empty or whitespace-only suggestion?** The backend already validates for blank output and treats it as a failure. An empty suggestion must not be stored; the absence of a suggestion is preferable to storing empty content.
- **What if multiple concurrent saves for the same session occur?** Suggestions are immutable once stored; the first stored suggestion wins. Subsequent generation attempts must be silently dropped if a suggestion already exists.
- **What if the session history item list loads before the suggestion is generated?** The list item should not expose or require the suggestion field — it is only visible in the session detail view.
- **How does the system handle suggestion display for sessions created before this feature?** Gracefully — the suggestion field is nullable; the UI renders nothing in its absence.

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: The system MUST generate exactly one AI coaching suggestion per saved program session.
- **FR-002**: The AI suggestion MUST be persisted in storage separately from user-entered data (session notes, feelings), linked to the session by the session identifier.
- **FR-003**: Once an AI suggestion has been stored for a session, it MUST NOT be changed or replaced — it is immutable.
- **FR-004**: The frontend MUST display the AI suggestion to the user immediately after the session is saved, in a clearly labelled section distinct from the user's own input.
- **FR-005**: The frontend MUST show a loading state in the suggestion area while awaiting the AI response, so the user understands content is being fetched.
- **FR-006**: If the AI suggestion cannot be generated (service failure, timeout, blank output), the user MUST still be able to view their saved session without any blocking error; the suggestion area MUST either show a neutral fallback message or be absent.
- **FR-007**: The AI suggestion MUST be available and displayed in the session detail view when the user revisits a session from the history.
- **FR-008**: Free (non-program) sessions MUST NOT trigger AI suggestion generation, and MUST NOT display a suggestion section.
- **FR-009**: The session history list view MUST NOT display or depend on the suggestion field — suggestions are only surfaced in the session detail view.
- **FR-010**: The backend API MUST expose the AI suggestion as part of the session detail response payload, in a dedicated field separate from user inputs.

### Technology Constraints *(mandatory)*

- **TC-001**: Backend services MUST use Java 21.
- **TC-002**: Backend services MUST use Spring Boot conventions and best practices.
- **TC-003**: Web frontend MUST use React.
- **TC-004**: Design MUST prefer simple, explicit logic over clever abstractions.
- **TC-005**: Database schema changes MUST use Flyway versioned migration scripts following the existing `VNNN__description.sql` naming convention.
- **TC-006**: The AI suggestion field in the `logged_sessions` table (or a related table) MUST be nullable to support sessions created before this feature.

### Key Entities *(include if feature involves data)*

- **Session AI Suggestion**: A single piece of text produced by the AI coaching model for a given program session. Attributes: session identifier (FK to `logged_sessions`), suggestion text (long text), generated-at timestamp. It is created at most once per session and never updated.
- **LoggedSession** (existing): A user's recorded workout. Will gain a navigable relationship to `SessionAiSuggestion` (one-to-one, optional).
- **SessionHistoryItem** (existing API DTO): Summary row in the history list — must NOT include the suggestion field.
- **LoggedSessionDetail** (existing API DTO): Full session details — must include the suggestion field (nullable string).

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: After saving a program session, users see an AI suggestion within 15 seconds in at least 95% of successful cases (where AI service is reachable).
- **SC-002**: 100% of persisted AI suggestions are immutable — no stored suggestion is ever overwritten by a subsequent generation attempt.
- **SC-003**: Users can return to any program session in their history and find the same suggestion text they saw at save time, with no data loss.
- **SC-004**: Sessions saved before this feature and free sessions display no suggestion section — zero regression errors in those cases.
- **SC-005**: When the AI service is unavailable, the session save flow completes successfully within the normal save time; the user is not blocked.

### UX Consistency Outcomes *(mandatory when UI changes exist)*

- **UX-001**: The AI suggestion section uses the same card/panel visual pattern as other informational sections on the session detail screen.
- **UX-002**: The section is labelled clearly (e.g., "AI Coaching Insight" or equivalent) so users understand it is AI-generated and not their own input.
- **UX-003**: The loading indicator matches the existing loading patterns used elsewhere in the application.

### Additional Non-Functional Outcomes *(include when materially relevant)*

- **NF-001**: AI suggestion generation MUST be asynchronous and MUST NOT increase the session save response time perceived by the user.
- **NF-002**: The suggestion storage MUST be decoupled from the session save transaction — a failure to store the suggestion MUST NOT roll back the session save.

## Assumptions

- The AI suggestion is only meaningful for program sessions because the AI model compares actual performance against program targets. Free sessions have no targets and no suggestion is generated.
- The suggestion is fetched by the frontend via a polling or separate GET endpoint after the session save response is received, rather than being included in the save (POST) response — since the AI generation is asynchronous and may complete after the POST returns.
- The suggestion text is plain text (no rich markdown rendering required in v1). If markdown is returned, it is displayed as-is.
- Users are authenticated before saving sessions; the suggestion is scoped to the authenticated user's session.
- The backend AI model (`LangChain4j` / `LangChainSessionProcessor`) is already integrated and operational; this feature only wires its output to persistence and the API response.
- Mobile-specific UI support is out of scope for this feature; the existing responsive layout is considered sufficient.
- There is no requirement to allow users to dismiss, rate, or interact with the AI suggestion beyond reading it.
- Notifications (push, email) for the suggestion are out of scope.
